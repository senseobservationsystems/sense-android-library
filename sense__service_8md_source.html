1 # Sense Service
2 
3 This is the main `SenseService` class to which application can bind.
4 
5 Activities or other components can not access the SenseService directly, but must do so through an nl.sense_os.service.SenseServiceStub instance, that can be gathered from the nl.sense_os.service.SenseService.SenseBinder instance referred in `onServiceConnected` callback of `ServiceConnection` defined when binding the service (see [Binding](#binding)). Through this `SenseServiceStub` instance, the application can then retrieve some functionality:
6 * Register user
7 * Login/Logout
8 * Toggle (start/stop) main sensing
9 * Toggle (start/stop) individual sensor
10 * Get/Set settings preferences
11 
12 `SenseService` is implemented as a singleton, and it has an instance of nl.sense_os.service.ServiceStateHelper to keep its state. It also has an instance of nl.sense_os.service.subscription.SubscriptionManager to manage all available sensors in a producer-subscriber model (see [Data Subcription](documentation/subscription.md)).
13 
14 `SenseService` should always be running as a foreground service, and should not be killed except by being stopped explicitly. There is also another component, nl.sense_os.service.AliveChecker (see [Alive Checker](#alive_checker)), that will check the service periodically and make sure it keeps running.
15 
16 `SenseService` is implemented in nl.sense_os.service.SenseService.

18 # Binding {#binding}
19 
20 `SenseService` is a service that others components (applications) can bind to. It will return an nl.sense_os.service.SenseService.SenseBinder object to the bound component through a derivative of android.content.ServiceConnection when connected. Through this binder, the component could get a `SenseServiceStub` object which provides an interface for `SenseService` functionality.
21 
22 Here is an example from [SensePlatform](documentation/sense_platform.md) on how to bind to `SenseService`.
23 
24 ~~~java
25 final Intent serviceIntent = new Intent(getContext(), nl.sense_os.service.SenseService.class);
26 boolean bindResult = mContext.bindService(serviceIntent, mServiceConnection, Context.BIND_AUTO_CREATE);
27 ~~~
28 
29 And here is an example of onServiceConnected callback of the ServiceConnection class that will receive the binder when connected to the service (also from SensePlatform).
30 
31 ~~~java
32 @Override
33 public void onServiceConnected(ComponentName className, IBinder binder) {
34  Log.v(TAG, "Bound to Sense Platform service...");
35 
36  mSenseService = ((SenseBinder) binder).getService();
37 }
38 ~~~
39 
40 Some of the functionality in `SenseServiceStub` provides a callback mechanism implementing an AIDL interface defined in nl.sense_os.service.ISenseServiceCallback. This interface provides several callback functions:
41 * `statusReport`, returns the current status of SenseService
42 * `onChangeLoginResult`, called when login status changes
43 * `onRegisterResult`, called when finished registering a user
44 
45 This is an example of how to use the callback mechanism in the login case.
46 
47 ~~~java
48 mSenseService.changeLogin(username, SenseApi.hashPassword(password), this.mServiceCallback);
49 ~~~

52 
53 ~~~
54 private final ISenseServiceCallback mServiceCallback = new ISenseServiceCallback.Stub() {
55  @Override
56  public IBinder asBinder() {
57  return null;
58  }
59 
60  @Override
61  public void onChangeLoginResult(final int result) throws RemoteException {
62 
63  // 0 if login completed successfully, -2 if login was forbidden,
64  // and -1 for any other errors.
65 
66  if (result == -2) {
67  onLoginFailure(LoginActivityFrag.this.ERROR_CREDENTIAL);
68 
69  } else if (result == -1) {
70  onLoginFailure(LoginActivityFrag.this.ERROR_CREDENTIAL);
71 
72  } else {

74  }
75  }

79 # Manage Main and Sensor Groups {#manage_main_sensor_groups}
80 
81 `SenseService` provides several methods to toggle the main state as well as sensor group states (see [Sensors](documentation/sensors.md)). There is a toggle method for each of the sensor groups. Here is an example of how to toggle the states of several sensor groups and the main service.
82 
83 ~~~
84 // enable some specific sensor modules
85 senseService.togglePhoneState(true);
86 senseService.toggleAmbience(true);
87 senseService.toggleMotion(true);
88 senseService.toggleLocation(true);
89 
90 // enable main service state
91 senseService.toggleMain(true);
92 ~~~

94 # Background service {#background_service}
95 
96 The `SenseService` will start as a service when nl.sense_os.service.SenseService.toggleMain method is called with true argument. It will also register itself as a foreground service so it doesn’t easily get killed by Android when the system is low on memory. It will also be started as a STICKY service so android knows to start it again if it’s accidentally been killed.
97 
98 When the service starts, it will try to start available sensors based on preferences (see [Settings](documentation/settings.md)). It will also start scheduling for data transmission to CommonSense. Finally it will send a broadcast so any other component will be notified that the `SenseService` has started (see [Start Broadcast](#start_broadcast)).
99 
100 Each sensor group will also be run in a different thread, and will keep existing while the service running.
101 
102 # Start Broadcast {#start_broadcast}
103 
104 `SenseService` will broadcast an *action_sense_service_broadcast* action everytime :
105 * The service started
106 * Sensor modules started/stop
107 
108 The broadcast receiver then can check the service status through the nl.sense_os.service.ServiceStateHelper instance.

110 # AliveChecker {#alive_checker}
111 
112 `SenseService` has an instance of nl.sense_os.service.AliveChecker to make sure that it keeps running when it should be. It will start checking periodically when `SenseService` starts sensing.
113 
114 `AliveChecker` will check every 15 minutes if the phone is awake, and it will make sure to wake up every hour. It will make sure the `SenseService` keeps running by trying to start the service regardless of current state.
